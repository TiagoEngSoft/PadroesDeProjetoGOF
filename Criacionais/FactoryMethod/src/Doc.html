🏭 Entendendo o Padrão Factory Method

Um exemplo simples com cartões personalizados

✅ Definição formal do padrão Factory Method (GOF)

Factory Method é um padrão de projeto criacional que define uma interface para a criação de objetos, mas permite que as subclasses decidam qual classe concreta será instanciada.

Ele promove o desacoplamento entre o código cliente e as classes concretas, permitindo maior flexibilidade e extensibilidade no sistema.

🤔 O que é Factory Method de forma simples?

É um padrão de projeto que define um **método de criação de objetos** na superclasse, mas permite que as **subclasses escolham qual objeto criar**.

Ou seja: ele separa a criação do objeto do seu uso.

💡 Exemplo prático

Imagine que você está criando um sistema para uma gráfica que imprime cartões.

Você pode ter cartões como:

  • 🎉 Cartão de Aniversário
  • 🎄 Cartão de Natal
  • 🙏 Cartão de Agradecimento
O sistema precisa gerar esses cartões automaticamente, mas **sem que o código principal saiba qual tipo está sendo criado**. Quem decide isso são as subclasses!

🧪 Exemplo de Código

1. Interface do Cartão

public interface Cartao {
    void exibirMensagem();
}

2. Implementações Concretas

public class CartaoAniversario implements Cartao {
    public void exibirMensagem() {
        System.out.println("Feliz Aniversário!");
    }
}

public class CartaoNatal implements Cartao {
    public void exibirMensagem() {
        System.out.println("Feliz Natal!");
    }
}

3. Criador Abstrato (Factory Method)

public abstract class CriadorCartao {
    // Factory Method
    public abstract Cartao criarCartao();

    // Método que usa o Cartão
    public void enviarCartao() {
        Cartao cartao = criarCartao();
        cartao.exibirMensagem();
    }
}

4. Criadores Concretos

public class CriadorCartaoAniversario extends CriadorCartao {
    public Cartao criarCartao() {
        return new CartaoAniversario();
    }
}

public class CriadorCartaoNatal extends CriadorCartao {
    public Cartao criarCartao() {
        return new CartaoNatal();
    }
}

5. Código Cliente

public class Main {
    public static void main(String[] args) {
        CriadorCartao criador1 = new CriadorCartaoAniversario();
        CriadorCartao criador2 = new CriadorCartaoNatal();

        criador1.enviarCartao(); // Saída: Feliz Aniversário!
        criador2.enviarCartao(); // Saída: Feliz Natal!
    }
}

✅ Por que usar esse padrão?

  • 📦 O código principal (cliente) não precisa saber qual classe está sendo usada.
    No nosso exemplo, o código cliente trabalha com a classe abstrata CriadorCartao e a interface Cartao, sem saber se está usando CartaoAniversario ou CartaoNatal. A criação do objeto acontece dentro do método criarCartao(), que é implementado nas subclasses (CriadorCartaoAniversario e CriadorCartaoNatal). Assim, o cliente não precisa mudar se quisermos criar um novo tipo de cartão.
  • 🔁 Fica fácil adicionar novos tipos de cartões sem mudar o código existente.
    Para criar um novo tipo, por exemplo CartaoAgradecimento, basta criar uma nova classe concreta do cartão, e uma nova subclasse do criador, como CriadorCartaoAgradecimento, implementando o método criarCartao() para retornar esse novo cartão. O cliente continua usando a interface CriadorCartao e não precisa ser alterado.
  • 🧼 Separa a lógica de criação da lógica de uso (desacoplamento).
    O método criarCartao() encapsula a criação do objeto, enquanto o método enviarCartao() na classe CriadorCartao usa o cartão criado. Essa separação facilita a manutenção e evolução do sistema, pois mudar como os cartões são criados não impacta o código que os usa.

📌 Conclusão

Factory Method é como um "modelo de produção".

Você define um processo (como criar), mas deixa que as subclasses decidam **o que será criado**.

Assim, o sistema continua organizado, flexível e fácil de expandir.
