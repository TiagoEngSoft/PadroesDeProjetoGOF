🎯 Padrão de Projeto: Strategy

Troca dinâmica de algoritmo (ou “estratégia”) para resolver um problema

✅ Definição formal do padrão Strategy (GOF)

O padrão Strategy define uma família de algoritmos (estratégias), encapsula cada um deles e os torna intercambiáveis. Strategy permite que algoritmos variem independentemente dos clientes que os utilizam.

🤔 Explicação simples

Imagine uma loja online. Para cada cliente, você pode aplicar diferentes tipos de desconto:
  • Sem desconto
  • Desconto fixo de 10%
  • Desconto fixo de 25%
Em vez de usar `if` ou `switch` para cada tipo de desconto no pedido, você seta uma “estratégia de desconto” e o pedido usa essa estratégia para calcular o valor final. Isso torna o código flexível e extensível.

💡 Nosso Exemplo: Estratégias de Desconto

  • DescontoStrategy: interface comum para todas as estratégias
  • DescontoFixo: aplica um percentual fixo de desconto
  • SemDesconto: não aplica desconto (estratégia padrão)
  • Pedido: classe que contém um preço e uma estratégia de desconto
  • Main: uso prático trocando estratégias

📦 Estrutura do Projeto


src/
├── strategy/
│   ├── DescontoStrategy.java
│   ├── DescontoFixo.java
│   ├── SemDesconto.java
│   └── Pedido.java
└── Main.java

🧪 Código Exemplo

DescontoStrategy.java

package strategy;

public interface DescontoStrategy {
    double calcular(double precoOriginal);
}

DescontoFixo.java

package strategy;

public class DescontoFixo implements DescontoStrategy {
    private double percentual;

    public DescontoFixo(double percentual) {
        this.percentual = percentual;
    }

    @Override
    public double calcular(double precoOriginal) {
        return precoOriginal * (1 - percentual);
    }
}

SemDesconto.java

package strategy;

public class SemDesconto implements DescontoStrategy {
    @Override
    public double calcular(double precoOriginal) {
        return precoOriginal;
    }
}

Pedido.java

package strategy;

public class Pedido {
    private double precoOriginal;
    private DescontoStrategy estrategia;

    public Pedido(double precoOriginal, DescontoStrategy estrategia) {
        this.precoOriginal = precoOriginal;
        this.estrategia = estrategia;
    }

    public double getPrecoFinal() {
        return estrategia.calcular(precoOriginal);
    }

    public void setEstrategia(DescontoStrategy novaEstrategia) {
        this.estrategia = novaEstrategia;
    }
}

Main.java

import strategy.*;

public class Main {
    public static void main(String[] args) {
        Pedido pedido = new Pedido(100.0, new SemDesconto());

        System.out.println("🧾 Preço original: R$100.00");
        System.out.println("🔸 Sem desconto: R$" + pedido.getPrecoFinal());

        pedido.setEstrategia(new DescontoFixo(0.1));
        System.out.println("🔸 Desconto de 10%: R$" + pedido.getPrecoFinal());

        pedido.setEstrategia(new DescontoFixo(0.25));
        System.out.println("🔸 Desconto de 25%: R$" + pedido.getPrecoFinal());
    }
}

🧠 Explicando o Código Passo a Passo

Agora vamos dissecar cada parte do código para você **entender exatamente o que acontece**:

  • 🔍 DescontoStrategy (interface):
    Essa interface define o método que toda estratégia de desconto deve implementar: double calcular(double precoOriginal).
    Isso garante que qualquer classe que implemente **DescontoStrategy** poderá ser usada de forma intercambiável.

  • 💵 DescontoFixo:
    - Recebe um valor `percentual` no construtor (por exemplo, 0,10 para 10%).
    - No método `calcular()`, ele aplica: `preçoOriginal * (1 - percentual)`.
    Ou seja, se o percentual for 0,10 (10%), ele retorna 90% do valor original.
    Isso representa uma lógica de desconto fixa configurável.

  • 🚫 SemDesconto:
    - Esta estratégia não aplica desconto algum.
    - O método `calcular()` simplesmente retorna `precoOriginal`.
    É útil como estratégia padrão ou “fallback” quando nenhum desconto deve ser aplicado.

  • 🛒 Pedido:
    Essa classe representa o cliente (o “contexto”) que usa uma estratégia de desconto.
    - **atributos**:
    • precoOriginal: o valor sem desconto
    • estrategia: a estratégia atual aplicada
    - **método `getPrecoFinal()`:** delega para a estratégia atual fazer o cálculo.
    - **método `setEstrategia(...)`:** permite trocar a estratégia de desconto em tempo de execução.

  • 🚀 Main.java:
    Aqui fazemos testes práticos:
    1. Criamos um pedido com valor 100, usando `SemDesconto` como estratégia inicial.
    2. Imprimimos o valor (100, sem desconto).
    3. Depois mudamos a estratégia para `DescontoFixo(0.1)` — desconto de 10% — e mostramos o valor (90).
    4. Por fim, trocamos para `DescontoFixo(0.25)` — desconto de 25% — e mostramdo (75).
    Observe que em nenhum momento usamos `if` ou `switch` para aplicar o desconto. O pedido delega tudo à estratégia escolhida.

Resumo visual:

  • Cliente (Pedido) → invoca getPrecoFinal()
  • Pedido → delega calcular() para a estratégia configurada
  • Estratégia → execução específica do cálculo
  • Podemos trocar estratégia durante a execução com setEstrategia(...)

✅ Benefícios do Strategy

  • 🔄 Código mais flexível — estratégias podem ser alteradas em tempo de execução
  • 🧩 Segregação de responsabilidades — cada cálculo de desconto fica em sua classe
  • 🛠 Facilita adicionar novas estratégias sem mexer no código existente (princípio Open/Closed)
  • 🔁 Evita estruturas condicionais pesadas

⚠️ Cuidados

  • 📂 Muitas classes se existirem muitas estratégias
  • 🧪 Teste bem as estratégias e as trocas em tempo de execução
  • ⚠️ Certifique-se de que a interface das estratégias é genérica e útil

📌 Conclusão

O padrão Strategy nos dá uma forma elegante de selecionar algoritmos comportamentais dinamicamente.

No nosso exemplo de descontos, ele nos permite trocar a forma de cálculo facilmente e com pouco acoplamento — ideal para sistemas que precisam ser expansíveis e flexíveis.
